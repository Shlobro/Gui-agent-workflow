# gui Developer Guide

## Purpose
Implements the interactive Qt UI for composing and running LLM workflows.

## Contents
- `main_window.py`: Main shell (File menu, toolbar, status bar, clear action) and `WorkflowCanvas` host.
- `canvas.py`: `QGraphicsView` orchestration, pan/zoom controls, connection creation, trigger-driven run execution with permanent Start node, pre-run validation, draw-time graph warnings, workflow save/load serialization, in-memory node clipboard (Ctrl+C / Ctrl+V), and undo/redo stack (Ctrl+Z / Ctrl+Y).
- `llm_node.py`: `WorkflowNode` — abstract base class shared by `LLMNode` and `FileOpNode`; owns connection bookkeeping, port proximity tests, `itemChange`, `boundingRect`, and the shared port-direction-arrow paint helper. `LLMNode(WorkflowNode)` — draggable LLM-call node with a painted dark-blue header strip ("LLM CALL"), embedded editor, model selector, and output panel. The proxy widget starts at y=28 (below the header). `StartNode` — permanent pure-trigger root node with output port only. `_GlowAnimator` singleton drives the animated "running" border effect (~33 fps QTimer, shared across running `LLMNode` instances).
- `llm_widget.py`: `LLMWidget` (embedded QWidget inside LLMNode with Name label, Model selector, Prompt area, and collapsible Output area), `ModelSelector` (viewport-anchored dropdown), and `_ModelListWidget`. `ModelSelector` emits `model_changed(old_id, new_id)` when the user picks a different model.
- `file_op_node.py`: `FileOpWidget`, `FileOpNode` base class, `CreateFileNode`, `TruncateFileNode`, `DeleteFileNode`. File-operation nodes that execute synchronously (no LLM subprocess) when triggered by the canvas. The painted header strip (y=0..28) shows the op-type label; the embedded `FileOpWidget` proxy is offset to y=28 so the user-editable Name field and Filename field appear below the header without overlap.
- `connection_item.py`: Obstacle-routed directed edge between workflow node ports.
- `undo_commands.py`: `AddNodeCommand`, `RemoveNodeCommand`, `MoveNodeCommand`, `AddConnectionCommand`, `RemoveConnectionCommand`, `TitleChangeCommand`, `ModelChangeCommand`, `PasteCommand` — covers every graph mutation.
- `project_chooser.py`: Startup dialog for selecting the project folder; persists recently used folders in `.recent_folders.json` at the repo root.
- `assets/`: Static logo files used by the model selector.
- `__init__.py`: Package marker.

## Key Interactions
- `WorkflowCanvas` owns node/edge lifecycle and triggers trigger-driven execution: for **Run All**, the permanent `StartNode` is the single root — its direct children fire first. LLM nodes execute via `LLMWorker` (exec_id-tagged); file-op nodes execute synchronously. On successful completion, downstream triggers are queued via the Qt event loop (`QTimer.singleShot(0, ...)`) to avoid deep synchronous recursion in cyclic file-op-only graphs, and each deferred callback re-checks that the source→target edge still exists before triggering. Run drain completes only after both active invocations and queued child-trigger callbacks are exhausted. Cycles still run until `stop_all()` is called. Run Selected and Run From Here bypass the Start node and use the selection as the entry point instead.
- Before any run, all reachable nodes are validated with node-type rules: LLMNode requires non-empty prompt and model; FileOpNode requires non-empty filename.
- `WorkflowCanvas` creates `LLMWorker` instances for `LLMNode`s and runs file operations synchronously for `FileOpNode`s.
- `LLMNode` holds user-editable metadata (title, prompt, model) used by execution and save/load.
- `MainWindow` accepts an optional `project_folder` string at construction, passes it to `WorkflowCanvas.set_working_directory()`, and delegates all graph behavior to the canvas. The **File** menu hosts Save Workflow, Load Workflow, and Open Project Folder actions; the toolbar no longer contains these.
- `ProjectChooserDialog` is shown at startup before the main window opens; the chosen folder is forwarded to `MainWindow`. The user can cancel the chooser and still open the window without a project folder, then pick one later via File → Open Project Folder.

## Behavior Notes
- The Start node is permanent: it cannot be deleted with the Delete key and is recreated after Clear Canvas. Its position is saved and restored with the workflow JSON.
- Cyclic graphs execute continuously: each node triggers its children on successful completion, so A→B→A loops until the user stops the workflow. Drawing a connection that would create a cycle shows a warning dialog at draw time, but the user may proceed and add it anyway.
- Nodes not reachable from Start are never executed by Run All.
- Children fire only on successful completion; an error node is a dead end and does not trigger its descendants.
- Drawing a second outgoing connection from a node that already has one shows an informational dialog explaining that the node will fan out to multiple parallel children; the user may proceed.
- Three run modes are available: **Run All** executes all nodes reachable from the permanent Start node (requires at least one connection from Start); **Run Selected** fires only the currently selected node(s) simultaneously without triggering their children; **Run From Here** fires the single selected node and all its descendants reachable from it. The Start-root restriction (reachability check and upfront validation) applies only to Run All; the other two modes validate only the nodes they will execute.
- Mouse wheel zoom is active on the canvas (zoom under cursor) with zoom clamped to prevent extreme scales, except while a model dropdown is open where wheel input scrolls the dropdown list instead of zooming.
- Canvas panning is driven by right-button drag (middle-button drag remains accepted).
- Left-drag on empty canvas uses a rubber-band rectangle to select multiple nodes at once.
- Background grid dots are rendered with a cosmetic pen so they stay visible and keep a fixed on-screen size across zoom levels.
- Scene bounds auto-expand around moved/loaded nodes with padding so corner nodes remain reachable after zooming and panning.
- Connection paths are routed around node bounding boxes (LLMNode, StartNode, FileOpNode) using obstacle-aware grid routing; route recompute runs immediately after connection creation and whenever nodes move/resize so lines avoid passing under nodes.
- Arrowheads terminate just outside the target input port, keeping direction visible instead of being hidden by the port dot.
- Node ports are direction-cued and color-differentiated: a left-side ingress arrow and right-side egress arrow protrude from the node edge to indicate flow, and each arrow hides once that side has at least one connection.
- Connection items use an expanded hit-test shape (line + arrowhead) so selecting arrows for delete is easier without rendering thicker lines.
- Selected nodes render an additional light-blue neon frame so active selection is visually obvious.
- `LLMNode` running state (`status == "running"`) displays an animated border driven by `_GlowAnimator` (pulsing halo + rotating sweep). `FileOpNode` uses its own paint-path glow math and does not register with `_GlowAnimator`.
- Deleting a node that has in-flight workers cancels those workers and moves their exec_ids to `_retired_exec_ids`. The `_active_workers` ref is kept alive until the callback fires (preventing a "QThread destroyed while still running" crash). When the callback arrives, `_on_invocation_done` detects the retirement flag, skips node mutation, and still calls `_check_drain()` so the run state transitions correctly if no other workers remain.
- Connections only define execution order; node outputs are displayed per node and are not injected into downstream prompts.
- **Copy/paste**: Ctrl+C snapshots all selected workflow nodes (`LLMNode` + `FileOpNode`, Start excluded) and any connections whose both endpoints are in the selection into an in-memory clipboard (`_clipboard` / `_clipboard_conns`), and resets a `_paste_count` counter to 0. Ctrl+V pushes a `PasteCommand` onto the undo stack; the command creates each node with a fresh `uuid4` node ID and a pre-assigned `_node_counter` label index; position is the original coordinates plus `40 * _paste_count` on both axes (incremented each paste, so successive pastes of the same clipboard land at distinct cascading offsets). Internal connections are recreated between the new nodes. Pasted nodes are selected and originals are deselected. Both shortcuts are suppressed when any embedded node widget has focus (via `QGraphicsProxyWidget` scene focus) so normal text editing is unaffected. The clipboard persists only for the session and is not saved with workflow JSON.
- **Undo/redo**: Every graph mutation (add node, delete node, add connection, delete connection, drag move, rename title, change model, paste) is wrapped in a `QUndoCommand` and pushed onto `WorkflowCanvas._undo_stack` (limit 100). Ctrl+Z calls `undo()` and Ctrl+Y calls `redo()` — both shortcuts are suppressed when any `QGraphicsProxyWidget` has scene focus (i.e. the user is interacting with any embedded node widget, including the prompt editor, title field, or model selector), so native widget undo still works inside text fields. Toolbar Undo/Redo buttons show the current command name and are enabled/disabled automatically by Qt, using node-type-specific labels for add/delete/rename node commands (LLM Node vs file-op node types). Multi-item deletes and multi-node drags are wrapped in a macro so they undo as one step. `_undo_stack.clear()` is called on `clear_canvas()` and `load_workflow_data()` so stale history cannot refer to deleted nodes. The `_undo_in_progress` flag prevents re-entrancy when `undo()`/`redo()` programmatically sets model or title, which would otherwise push a new command. Raw (no-stack) node manipulation helpers (`_undo_add_node`, `_undo_remove_node`, `_undo_add_connection`, `_undo_remove_connection_item`) are called only by command objects.
- **File operation nodes** (`CreateFileNode`, `TruncateFileNode`, `DeleteFileNode`) are subclasses of `FileOpNode` defined in `file_op_node.py`. They embed a `FileOpWidget` (Name label + name field + Filename label + filename input + output area) positioned below the painted header strip so the two text areas do not overlap the op-type title. When executed, `canvas._fire_file_op` runs the operation synchronously on the main thread (no worker thread), writes success/error text into the FileOp output pane, and then completes via `_on_invocation_done`. Validation requires a non-empty `filename` instead of a non-empty prompt and model. Serialization includes a `node_type` field (`"create_file"` / `"truncate_file"` / `"delete_file"`) so `load_workflow_data` can reconstruct the correct class via `NODE_TYPE_MAP`. Paths are resolved relative to the active project folder and rejected if absolute or outside that folder after canonicalization. `create_file` reports `Already exists` when the target file already exists and errors if the existing target is a directory/non-file path.
- Model selection uses a compact selector row that opens an overlay dropdown on the canvas viewport (not a Qt popup window), keeps node height unchanged while open, anchors dropdown position from scene coordinates so it stays aligned with moved/zoomed nodes, clamps geometry to visible viewport bounds (opening upward when needed), shows provider logos from `assets/` (Anthropic/OpenAI/Gemini), normalizes them to a 16x16 icon canvas, auto-selects the first available model, and lazy-loads provider modules if needed.
