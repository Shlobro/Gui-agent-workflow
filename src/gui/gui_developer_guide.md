# gui Developer Guide

## Purpose
Implements the interactive Qt UI for composing and running LLM workflows.

## Contents
- `main_window.py`: Main shell (File menu, toolbar, status bar). Sets `WorkflowCanvas` as the central widget and parents `PropertiesPanel` directly onto the canvas as a floating overlay, pinned to the top-right corner via `setGeometry` (driven by a canvas resize event filter and the panel animation's `valueChanged` signal). This overlay approach prevents the canvas viewport from shrinking when the panel opens. Drives the panel by listening to `canvas.selection_changed` and wiring panel signals to canvas undo handlers. Run and save actions call `PropertiesPanel.commit_pending_edits()` before delegating to canvas execution/serialization paths. Load/Clear hide the panel before destructive canvas mutations so stale panel commits cannot target replaced/deleted nodes. Load catches parse/schema errors and shows a user-facing error dialog instead of crashing. Listens to `canvas.usage_limit_hit(node_id, error_text)` and opens `UsageLimitDialog`; if the user chooses **Change Model**, the errored node is selected and the properties panel slides open automatically.
- `dialogs/` subpackage: Modal dialog classes for runtime user notifications.
  - `dialogs/usage_limit_dialog.py`: `UsageLimitDialog(QDialog)` — shown when a usage/rate-limit error is detected. See `dialogs/dialogs_developer_guide.md`.
- `canvas/` subpackage: houses `WorkflowCanvas` and its two mixins.
  - `canvas/__init__.py`: `WorkflowCanvas(QGraphicsView)` — orchestration class that inherits `_ExecutionMixin` and `_IOMixin`. Owns pan/zoom controls, connection drawing, keyboard shortcuts, node add/remove, and the undo/redo stack (Ctrl+Z / Ctrl+Y). Exposes `on_output_line` and `on_output_cleared` callback attributes set by `MainWindow` to forward node output into the panel. Exposes `notify_node_changed(node_id)` so undo/redo of title, model, file-op type, loop count, or git action type changes causes the panel to refresh.
  - `canvas/execution.py`: `_ExecutionMixin` — run/stop logic: `run_all`, `run_selected_only`, `run_from_here`, `stop_all`, graph traversal, pre-run validation, the invocation engine (`_fire_invocation`, `_fire_file_op`, `_fire_git_action`, `_fire_condition_check`, `_on_condition_done`, `_fire_loop`, `_on_loop_iteration`), deferred child triggering with lineage-token fan-out, and drain checking. `GitActionNode` invocations run through `GitWorker` so long git commands do not block the UI and are cancellable by `stop_all`; `GitWorker` validates cwd existence before launch and emits a dedicated cwd error when invalid. Each trigger call carries two tokens: `lineage_token` (UUID string, forks on fan-out for general thread isolation) and `loop_token` (UUID string, stable for the full lifetime of one loop thread so the counter key survives downstream fan-out). `LoopNode` counters in `canvas._loop_counters` are keyed by `(node_id, loop_token)` to isolate parallel loop threads.
  - `canvas/io.py`: `_IOMixin` — undo primitive helpers (`_undo_add_node`, `_undo_remove_node`, `_undo_add_connection`, `_undo_remove_connection_item`, `_find_connection`), clipboard (`_copy_selected`, `_paste`), `clear_canvas`, and `get_workflow_data` / `load_workflow_data` (delegates parse/validate to `workflow_io`, then clears/applies only on success; malformed node records abort the load, malformed connections are silently dropped).
- `llm_node.py`: `WorkflowNode` — abstract base class shared by `LLMNode`, `FileOpNode`, `ConditionalNode`, `LoopNode`, and `GitActionNode`; owns connection bookkeeping, port proximity tests, `itemChange`, `boundingRect`, and the shared port-direction-arrow paint helper. `LLMNode(WorkflowNode)` — compact painted node (fixed height 64 px) with a dark-blue "LLM CALL" header strip and a title label in the body area. All editing is done in `PropertiesPanel`, not inline. Stores `_title`, `_model_id`, and `_prompt_text` as plain Python attributes. `StartNode` — permanent pure-trigger root node with output port only. `_GlowAnimator` singleton drives the animated "running" border effect (~33 fps QTimer, shared across running `LLMNode` instances).
- `llm_widget.py`: `ModelSelector` (overlay dropdown widget), `_ModelListWidget`, `NODE_WIDTH`, `ICON_SIZE`, provider icon helpers (`provider_icon`, `_fallback_provider_icon`, `_normalized_logo_pixmap`, `provider_company`, `get_registered_providers`), and `populate_model_selector` — a helper that fills any `ModelSelector` with all registered provider models. Used by `PropertiesPanel`.
- `file_op_node.py`: `FileOpNode` — single concrete class for file operations. The operation type (`create_file`, `truncate_file`, `delete_file`) is stored as a mutable `node_type` instance attribute; the user changes it via the properties panel op-type dropdown. `CreateFileNode`, `TruncateFileNode`, `DeleteFileNode` are convenience factory functions (not subclasses) that pre-set `node_type` and are used by `NODE_TYPE_MAP` for load/paste. `NODE_TYPE_MAP` also includes `"conditional"`, `"loop"`, and `"git_action"` (via lazy import factories) so a single dict covers all node types. Compact painted nodes (fixed height 64 px) with an accent-colored header determined by `node_type`. Stores `_title` and `_filename` as plain attributes; editing is done in `PropertiesPanel`.
- `git_action_node.py`: `GitActionNode` — a compact node (64 px) for git operations. Stores `git_action` (`"git_add"`, `"git_commit"`, `"git_push"`), `msg_source` (`"static"` or `"from_file"`), `commit_msg`, `commit_msg_file`, and `_title` as plain attributes. Accent colors: `git_add` dark green, `git_commit` dark blue, `git_push` dark purple. Single input/output port at vertical center. `NODE_TYPE_MAP` wires git node creation through `file_op_node._git_action_factory` (lazy import), which returns `GitActionNode`.
- `_panel_forms.py`: Form widget classes (`_LLMForm`, `_FileOpForm`, `_ConditionalForm`, `_LoopForm`, `_GitActionForm`) used by `PropertiesPanel`. Split out to keep file sizes under 1000 lines. Imported by `properties_panel.py`.
- `properties_panel.py`: `PropertiesPanel(QWidget)` — animated slide-in drawer that opens when a node is selected and closes when nothing is selected. Contains `_LLMForm` (page 1), `_FileOpForm` (page 2), `_ConditionalForm` (page 3), `_LoopForm` (page 4), and `_GitActionForm` (page 5) inside scroll areas on a `QStackedWidget`. `_ConditionalForm` provides a condition-type dropdown (populated from `CONDITION_REGISTRY`), name field, filename field ("File to check"), and a read-only result area. `_LoopForm` provides a name field, an iterations `QSpinBox` (min 1, max 9999, default 3), and a read-only output area. `_GitActionForm` provides an action dropdown (`git_add`/`git_commit`/`git_push`), name field, and a commit-options sub-frame (hidden unless `git_commit` is selected) with message-source dropdown and static-text or file-path input. Animation is driven by `QPropertyAnimation` on `maximumWidth` (200 ms, OutCubic). Emits `title_committed`, `model_changed`, `prompt_committed`, `filename_committed`, `op_type_changed`, `condition_type_changed`, `loop_count_changed`, and `git_action_changed` signals. `_GitActionForm` also emits `msg_source_changed`; `PropertiesPanel` handles that internally by writing `GitActionNode.msg_source` directly (no undo command). Exposes `maybe_append_output` / `maybe_clear_output` for streaming output from the canvas and `refresh_if_current(node)` to force a field reload when needed. The `filename_committed` signal is reused for both `FileOpNode` and `ConditionalNode` filename fields; `MainWindow` routes it to the correct node type. `commit_msg` and `commit_msg_file` on `GitActionNode` are written directly to node attrs (no undo command), same pattern as `filename` on `FileOpNode`.
- `workflow_io.py`: Pure serialization/validation helpers: `parse_workflow_data` (validates and normalises a raw JSON dict — node errors abort, bad connections are dropped), `build_workflow_data` (serialises canvas state to a dict), and `get_provider_for_model` (registry lookup by model ID). No Qt widget state; imported by the `canvas/` subpackage.
- `conditional_node.py`: `ConditionalNode` — a node with two output ports (true/false) that evaluates a condition and routes execution to the matching branch. `CONDITION_REGISTRY` maps condition IDs to `(display_name, evaluator_fn)` entries; the only built-in condition is `"file_empty"`. Stores `_title`, `_filename`, and `condition_type` as plain attributes. Node height is 80 px (taller than compact nodes) to accommodate two labeled port circles. `is_near_true_port`/`is_near_false_port` provide per-port proximity tests; `output_port_scene_pos(port)` accepts `"true"`, `"false"`, or `"output"` (defaults to true port). New conditions can be added by registering them in `CONDITION_REGISTRY`.
- `loop_node.py`: `LoopNode` — a node with two output ports (loop/done) on the right edge. Each time it is triggered, it fires the **loop** port for iterations 1 through N-1 and fires the **done** port on iteration N. Loop counters are stored in `canvas._loop_counters: Dict[tuple, int]` keyed by `(node_id, loop_token)` so parallel loop threads each get an independent counter; the `loop_token` is stable for the full lifetime of one loop thread and is distinct from the `lineage_token` that forks on fan-out. After the done branch fires the counter resets automatically. Stores `_title` and `loop_count: int` (default 3, range 1–9999) as plain attributes. Node height is 80 px. `is_near_loop_port`/`is_near_done_port` provide per-port proximity tests; `output_port_scene_pos(port)` accepts `"loop"`, `"done"`, or `"output"` (defaults to loop port). Drawing a connection from the loop port back into the graph does not trigger the cycle-warning dialog.
- `connection_item.py`: Obstacle-routed directed edge between workflow node ports. Each `ConnectionItem` now carries a `source_port` attribute (default `"output"`); for `ConditionalNode` connections this is `"true"` or `"false"`. `to_dict()` includes `"source_port"` only when it differs from `"output"` for backward-compatible saves. `update_path()` passes `source_port` to `source_node.output_port_scene_pos()`.
- `undo_commands.py`: `AddNodeCommand`, `RemoveNodeCommand`, `MoveNodeCommand`, `AddConnectionCommand`, `RemoveConnectionCommand`, `TitleChangeCommand`, `ModelChangeCommand`, `FileOpTypeChangeCommand`, `ConditionTypeChangeCommand`, `LoopCountChangeCommand`, `GitActionTypeChangeCommand`, `PasteCommand` — covers every graph mutation. `AddConnectionCommand` and `RemoveConnectionCommand` each carry `source_port` to correctly target the right edge on multi-port nodes.
- `project_chooser.py`: Startup dialog for selecting the project folder; persists recently used folders in `.recent_folders.json` at the repo root.
- `assets/`: Static logo files used by the model selector.
- `__init__.py`: Package marker.

## Key Interactions
- `WorkflowCanvas` owns node/edge lifecycle and triggers trigger-driven execution: for **Run All**, the permanent `StartNode` is the single root — its direct children fire first. LLM nodes execute via `LLMWorker` (exec_id-tagged); git nodes execute via `GitWorker` (exec_id-tagged); file-op, conditional, and loop nodes execute synchronously. On successful completion, downstream triggers are queued via the Qt event loop (`QTimer.singleShot(0, ...)`) to avoid deep synchronous recursion in cyclic file-op-only graphs, and each deferred callback re-checks that the source→target edge still exists before triggering. Run drain completes only after both active invocations and queued child-trigger callbacks are exhausted. `stop_all()` cancels both LLM and git workers. Cycles still run until `stop_all()` is called. Run Selected and Run From Here bypass the Start node and use the selection as the entry point instead.
- Before any run, all reachable nodes are validated with node-type rules: LLMNode requires a non-empty prompt and a model ID that resolves to a registered provider model; FileOpNode requires non-empty filename; ConditionalNode requires non-empty filename and a `condition_type` present in `CONDITION_REGISTRY`; LoopNode has no extra validation requirement beyond `loop_count` always being a valid integer (enforced at parse/load time); GitActionNode requires `git_action` in `{git_add, git_commit, git_push}` and `msg_source` in `{static, from_file}`; for `git_commit`, a non-empty commit message is required (if `msg_source == "static"`) or a non-empty commit message file path is required (if `msg_source == "from_file"`).
- `MainWindow` wires `canvas.selection_changed` → `_on_selection_changed` which calls `_panel.show_for_node(node)` for a single selected non-Start node, or `_panel.hide_panel()` otherwise. Panel signals are wired to canvas undo handlers and direct node attribute setters.
- `LLMNode` stores user-editable metadata (`_title`, `_model_id`, `_prompt_text`) as plain Python attributes. `FileOpNode` stores user-editable metadata (`_title`, `node_type`, `_filename`) as plain Python attributes. `ConditionalNode` stores user-editable metadata (`_title`, `_filename`, `condition_type`) as plain Python attributes. `GitActionNode` stores user-editable metadata (`_title`, `git_action`, `msg_source`, `commit_msg`, `commit_msg_file`) as plain Python attributes. The canvas reads these during execution and serialization.
- `MainWindow` accepts an optional `project_folder` string at construction, passes it to `WorkflowCanvas.set_working_directory()`, and delegates all graph behavior to the canvas. The **File** menu hosts Save Workflow, Load Workflow, and Open Project Folder actions.
- `ProjectChooserDialog` is shown at startup before the main window opens; the chosen folder is forwarded to `MainWindow`. The user can cancel the chooser and still open the window without a project folder, then pick one later via File → Open Project Folder.

## Behavior Notes
- The Start node is permanent: it cannot be deleted with the Delete key and is recreated after Clear Canvas. Its position is saved and restored with the workflow JSON.
- Cyclic graphs execute continuously: each node triggers its children on successful completion, so A→B→A loops until the user stops the workflow. Drawing a connection that would create a cycle shows a warning dialog at draw time, but the user may proceed and add it anyway.
- Nodes not reachable from Start are never executed by Run All.
- Children fire only on successful completion; an error node is a dead end and does not trigger its descendants.
- Drawing a second outgoing connection from a node that already has one shows an informational dialog explaining that the node will fan out to multiple parallel children; the user may proceed.
- Three run modes are available: **Run All** executes all nodes reachable from the permanent Start node (requires at least one connection from Start); **Run Selected** fires only the currently selected node(s) simultaneously without triggering their children; **Run From Here** fires the single selected node and all its descendants reachable from it. The Start-root restriction (reachability check and upfront validation) applies only to Run All; the other two modes validate only the nodes they will execute.
- Mouse wheel zoom is active on the canvas (zoom under cursor) with zoom clamped to prevent extreme scales, except while a model dropdown is open where wheel input scrolls the dropdown list instead of zooming.
- Canvas panning is driven by right-button drag (middle-button drag remains accepted).
- Left-drag on empty canvas uses a rubber-band rectangle to select multiple nodes at once.
- Background grid dots are rendered with a cosmetic pen so they stay visible and keep a fixed on-screen size across zoom levels.
- Scene bounds auto-expand around moved/loaded nodes with padding so corner nodes remain reachable after zooming and panning.
- Connection paths are routed around node bounding boxes for `StartNode` plus every `WorkflowNode` subtype (`LLMNode`, `FileOpNode`, `ConditionalNode`, `LoopNode`, `GitActionNode`) using obstacle-aware grid routing; route recompute runs immediately after connection creation and whenever nodes move/resize so lines avoid passing under nodes.
- Arrowheads terminate just outside the target input port, keeping direction visible instead of being hidden by the port dot.
- Node ports are direction-cued and color-differentiated: a left-side ingress arrow and right-side egress arrow protrude from the node edge to indicate flow, and each arrow hides once that side has at least one connection.
- Connection items use an expanded hit-test shape (line + arrowhead) so selecting arrows for delete is easier without rendering thicker lines.
- Selected nodes render an additional light-blue neon frame so active selection is visually obvious.
- `LLMNode` running state (`status == "running"`) displays an animated border driven by `_GlowAnimator` (pulsing halo + rotating sweep). `FileOpNode` uses its own paint-path glow math and does not register with `_GlowAnimator`.
- Deleting a node that has in-flight workers cancels those workers and moves their exec_ids to `_retired_exec_ids`. The `_active_workers` ref is kept alive until the callback fires (preventing a "QThread destroyed while still running" crash). When the callback arrives, `_on_invocation_done` detects the retirement flag, skips node mutation, and still calls `_check_drain()` so the run state transitions correctly if no other workers remain.
- Connections only define execution order; node outputs are displayed in the properties panel when the node is selected, not injected into downstream prompts.
- **Copy/paste**: Ctrl+C snapshots all selected workflow nodes (`LLMNode`, `FileOpNode`, `ConditionalNode`, `LoopNode`, and `GitActionNode`, Start excluded) and any connections whose both endpoints are in the selection into an in-memory clipboard, and resets a `_paste_count` counter to 0. Ctrl+V pushes a `PasteCommand` onto the undo stack; the command creates each node with a fresh `uuid4` node ID and a pre-assigned `_node_counter` label index; position is the original coordinates plus `40 * _paste_count` on both axes (incremented each paste, so successive pastes of the same clipboard land at distinct cascading offsets). Internal connections are recreated between the new nodes. Pasted nodes are selected and originals are deselected. Both shortcuts are suppressed when any text widget (`QLineEdit`, `QTextEdit`, `QPlainTextEdit`) has focus so normal text editing is unaffected. The clipboard persists only for the session and is not saved with workflow JSON.
- **Undo/redo**: Every graph mutation (add node, delete node, add connection, delete connection, drag move, rename title, change model, change file-op type, change condition type, change loop count, change git action type, paste) is wrapped in a `QUndoCommand` and pushed onto `WorkflowCanvas._undo_stack` (limit 100). Ctrl+Z calls `undo()` and Ctrl+Y calls `redo()` — both shortcuts are suppressed when any text widget has focus. Toolbar Undo/Redo buttons show the current command name and are enabled/disabled automatically by Qt. Multi-item deletes and multi-node drags are wrapped in a macro so they undo as one step. `_undo_stack.clear()` is called on `clear_canvas()` and `load_workflow_data()` so stale history cannot refer to deleted nodes. After a title, model, file-op type, loop-count, or git-action-type undo/redo, `canvas.notify_node_changed(node_id)` re-emits `selection_changed` if the node is selected, causing `MainWindow._on_selection_changed` to call `_panel.show_for_node` with fresh data.
- **Conditional nodes** (`ConditionalNode`) have two output ports on the right edge: true (upper, ~33% height, green) and false (lower, ~67% height, red). The user draws connections from each port independently. During execution `_fire_condition_check` resolves the filename through the same project-folder confinement guard as file ops, evaluates the condition synchronously with the resolved absolute path, appends a `"true"` or `"false"` result line to the node output, then calls `_on_condition_done` which queues child triggers only on the matching port. Pre-run validation requires a non-empty filename and a `condition_type` present in `CONDITION_REGISTRY`; `parse_workflow_data` also rejects unknown `condition_type` values at load time. Serialization uses `node_type: "conditional"` plus `condition_type` and `filename` keys. Connections from a `ConditionalNode` carry `source_port: "true"` or `source_port: "false"` in their serialized form; the default `"output"` is omitted for backward compatibility. When loading connections, `source_port` is validated against node type: `"true"`/`"false"` for `ConditionalNode` sources, `"output"` for all others; connections with invalid `source_port` are silently dropped. Drawing a second connection from the same port of a `ConditionalNode` shows the parallel fan-out warning; drawing the first connection on the other (complementary) port does not.
- **Loop nodes** (`LoopNode`) have two output ports on the right edge: loop (upper, ~33% height, blue) and done (lower, ~67% height, green). The user draws connections from each port independently. During execution `_fire_loop` increments a counter keyed by `(node_id, loop_token)` in `canvas._loop_counters`; if the count is below `loop_count` it fires the loop port (queuing child triggers that eventually re-enter the LoopNode), otherwise it fires the done port and removes the counter. Each independent execution thread carries a stable `loop_token` (UUID assigned on first `_fire_loop` entry) so parallel threads hitting the same LoopNode maintain separate counters. When the loop branch has no outgoing connections, or when `_no_fanout` is set (Run Selected mode), the counter is immediately cleared and the node is set to done. On natural run completion (`_check_drain`), any remaining running nodes (including LoopNodes whose loop path never cycled back) are set to done and `_loop_counters` is cleared. Pre-run validation has no extra requirements for LoopNode beyond `loop_count` being a valid integer (1–9999), which is enforced at parse/load time by `parse_workflow_data` and clamped in `from_dict`. Serialization uses `node_type: "loop"` plus `loop_count` and `name` keys. Connections from a `LoopNode` carry `source_port: "loop"` or `source_port: "done"` in their serialized form; `"output"` is omitted for backward compatibility. When loading connections, `source_port` is validated against node type: `"loop"`/`"done"` for `LoopNode` sources; connections with invalid `source_port` are silently dropped. Drawing a connection from the loop port suppresses the cycle-warning dialog so loop-back edges can be created without prompts.
- **File operation nodes** use a single `FileOpNode` class whose `node_type` instance attribute (`"create_file"` / `"truncate_file"` / `"delete_file"`) determines behavior. The user creates any file-op node via the single **＋ File Op** toolbar button (defaults to `create_file`) and changes the operation type in the properties panel operation dropdown, which pushes a `FileOpTypeChangeCommand` onto the undo stack. When executed, `canvas._fire_file_op` runs the operation synchronously on the main thread (no worker thread), writes success/error text into `node.output_text`, and then completes via `_on_invocation_done`. Validation requires a non-empty `filename` instead of a non-empty prompt and model. Serialization includes a `node_type` field so `load_workflow_data` can reconstruct the correct type via `NODE_TYPE_MAP`. Paths are resolved relative to the active project folder and rejected if absolute or outside that folder after canonicalization. `create_file` reports `Already exists` when the target file already exists and errors if the existing target is a directory/non-file path.
- **Properties panel**: `PropertiesPanel` slides in from the right when a single non-Start node is selected. The drawer uses `QPropertyAnimation` on `maximumWidth` (0 → 360 px, 200 ms); `MainWindow` connects `_anim.valueChanged` to `_reposition_panel` so each animation frame calls `setGeometry` on the panel, translating the animated width into a real geometry change since the panel is an absolutely positioned overlay child of the canvas. Editing the name field and pressing Enter/Tab emits `title_committed` which pushes a `TitleChangeCommand`. Changing the model emits `model_changed` which pushes a `ModelChangeCommand`. For file-op nodes, changing the operation-type dropdown emits `op_type_changed` which pushes a `FileOpTypeChangeCommand`. For conditional nodes, changing the condition-type dropdown emits `condition_type_changed` which pushes a `ConditionTypeChangeCommand`. For git nodes, changing the action dropdown emits `git_action_changed` which pushes a `GitActionTypeChangeCommand`; changing message source writes `node.msg_source` directly with no undo entry. Prompt changes commit on prompt focus loss; filename changes commit on filename focus loss or Enter in the filename field; both write directly to node attributes (no undo entry). Git commit message fields commit on focus loss/Enter and write directly to node attrs (no undo entry). `commit_pending_edits()` commits visible-form edits without collapsing the panel (flushes content fields first, then commits title) and is used before run/save and when switching the panel to a different node, so pending edits are applied before execution, serialization, or panel reload. Output from running nodes streams into the output area of the panel in real time via `maybe_append_output`; for git nodes, streamed lines are preserved and only a fallback success line is appended when the git command is otherwise silent. The panel is cleared and collapsed when canvas is cleared or workflow is loaded.
- Model selection uses a compact selector row that opens an overlay dropdown anchored to the active window (or the canvas viewport when proxy-embedded), clamps geometry to visible bounds (opening upward when needed), shows provider logos from `assets/` (Anthropic/OpenAI/Gemini), normalizes them to a 16x16 icon canvas, supports an explicit "no selection" state (`current_model_id == None`), and lazy-loads provider modules if needed. New LLM nodes default to `gemini-3-pro-preview` when available in the provider registry; otherwise they use the first registered model.
